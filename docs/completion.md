- 只记录已经完成或明确决策的内容，不记录未确认的想法。

## 2026-06-14 抢票重试间隔改为毫秒

类型：配置调整

摘要：任务运行中的抢票重试间隔从秒级配置改为毫秒级配置，并同步更新 Web 表单提示。

主要变更：

- 任务模型新增 `pollIntervalMillis`，运行器按毫秒计算重试等待时间。

- Web 任务配置表单将“轮询间隔”改为“重试间隔（ms）”，默认值为 `1000`。
- 更新 `docs/api.md`，将任务请求/响应示例和下发说明改为 `pollIntervalMillis`。
- 新增测试覆盖毫秒字段持久化。

验收情况：

- 已通过 `go test ./...`。
- 已通过 `npm run build`。

遗留事项：

- 当前只支持用户手动填写毫秒值，尚未提供快捷档位或错误码级别退避策略。


## 2026-06-14 预热连接保持 keep-alive 复用

类型：功能增强

摘要：开票前预热请求会保留 keep-alive 空闲连接，后续抢票请求复用同一个票务 HTTP client 的连接池。

主要变更：

- 默认票务 HTTP client 使用显式 keep-alive Transport，配置空闲连接池和空闲超时时间。

- `docs/api.md` 补充说明预热连接会保留在空闲连接池中供订单请求复用。
- 新增测试验证两次预热 `HEAD` 后，紧接的订单准备请求复用同一条连接。

验收情况：

- 已通过 `go test ./...`。

遗留事项：

- 连接复用依赖目标服务端和 Go Transport 的正常 keep-alive 行为；若服务端主动关闭连接，后续请求会自动新建连接。

## 2026-06-14 任务运行重试机制更新

类型：运行逻辑调整

摘要：统一任务运行阶段的错误处理策略，运行中遇到接口错误时保持任务运行并继续重试。

主要变更：

- 票档检测、订单准备、订单创建阶段遇到接口错误时，任务保持 `running`，写入 `warn` 日志，并按任务配置的轮询间隔继续重试。
- `createV2` 返回 `code=0` 且消息包含 `defaultBBR` 时，作为可重试警告处理，不进入待支付状态。
- `createV2` 返回 `100034` 且携带有效 `pay_money` 时，自动更新任务金额后继续重试。
- 订单已创建但获取支付参数失败时，只重试支付参数获取，避免重复创建订单。
- 重试机制仍受任务暂停、取消和任务结束时间约束；超过结束时间后停止任务。
- `createV2` 错误提示保持标准格式：`状态码：%d，提示信息：...`。
- 本记录为当前重试机制的最终语义说明，覆盖此前关于 `defaultBBR` 是否成功的过渡性记录。

验收情况：

- 已通过 `go test ./...`。
- 已通过 `git diff --check`。

遗留事项：

- 当前重试间隔复用任务轮询间隔，尚未单独设计指数退避或错误码级别的重试策略。
- 合规边界不变：只按配置轮询间隔重试，不绕过验证码、不绕过风控、不承诺成功率。
