# Biligo 项目演进记录

本文档用于记录项目每一次重要演进。后续每次完成需求、阶段目标、验收、修复或架构调整时，都应追加一条记录，便于回溯项目状态和决策。

## 记录规范

- 按时间倒序追加记录，最新记录放在最上方。
- 每条记录包含：日期、类型、摘要、主要变更、验收情况、遗留事项。
- 只记录已经完成或明确决策的内容，不记录未确认的想法。
- 聊天中提到: “记录演进”，在将演进记录添加到本文件中，否则不添加

## 2026-06-20 并发代理支持

类型：功能增强

摘要：任务代理组新增循环代理和并发代理两种模式，并优化 API 代理组拉取完成日志，输出本次准备好的全部代理节点。

主要变更：

- 任务模型、存储和 API 增加 `proxyMode` 字段，支持 `round_robin` 循环代理和 `concurrent` 并发代理，默认保持循环代理。
- 循环代理保留原有行为：抢票阶段使用当前代理节点，遇到代理风控或网络请求错误时切换到下一个节点。
- 并发代理在抢票模式和抢票+回流捡漏模式的抢票阶段生效，每个可用代理节点启动一个抢票线程；任一线程成功进入支付或检测到重复订单后停止其他线程。
- 回流捡漏模式和抢票+回流捡漏模式的回流阶段仍不使用代理组。
- Web 任务配置中将代理设置整理为独立小区域，选择代理组后可选择循环代理或并发代理；任务列表和任务状态展示代理模式。
- API 代理组拉取完成日志从只输出当前节点，调整为输出本次准备好的全部代理节点，例如“已准备 2 个代理节点：host1:port、host2:port”。
- `docs/api.md` 补充 `proxyMode` 字段、两种代理模式行为和 API 代理组日志说明。

验收情况：

- 已新增存储、HTTP API 和运行器测试，覆盖 `proxyMode` 持久化、并发代理无代理组校验、并发代理成功/重复订单停止其他线程，以及 API 拉取日志输出全部节点。
- 已通过 `go test ./...`。
- 已通过 `pnpm --dir web build`。
- 已通过 `git diff --check`。

## 2026-06-20 开票前 hot_project 状态校验

类型：行为优化

摘要：抢票任务在开票前 5 分钟重新校验项目 `hot_project` 状态，降低任务创建后项目风控状态变化导致下单参数不匹配的风险。

主要变更：

- 抢票模式和抢票+回流捡漏模式在等待起售阶段，距离起售不超过 5 分钟时拉取一次项目详情并校验 `hot_project` 状态。
- 若远端 `hot_project` 与任务本地状态不一致，会更新任务 `is_hot_project` 并按新状态继续执行当前任务。
- 校验失败会最多重试 10 次；仍失败时记录警告日志，并继续使用任务本地状态抢票。
- 下单前不会循环反复校验；本次任务运行中仅在 5 分钟窗口触发一次。
- 新增任务 `is_hot_project` 单字段更新逻辑，并补充任务下发 API 文档说明。

验收情况：

- 已新增运行器测试，覆盖开票前状态变化后更新任务并使用 hot project 参数，以及连续失败 10 次后继续使用本地状态。

## 2026-06-18 订单创建重试策略调整

类型：行为优化

摘要：调整订单阶段重试策略，改为每请求 1 次订单准备 `prepare`，最多连续请求 4 次 `createV2` 创建订单。

主要变更：

- 抢票模式、回流捡漏模式、抢票+回流捡漏模式统一使用“1 次 prepare + 4 次 createV2”策略。
- `createV2` 在前 3 次失败时会继续复用同一次 prepare 的结果重试，第 4 次仍失败后才进入下一轮等待与重新 prepare。
- `createV2` 返回 `100034` 时仍会自动更新订单金额，并在同一轮 createV2 重试中使用新金额继续创建订单。
- 回流捡漏命中可购买票种后，只有同一次 prepare 下的 4 次 createV2 都失败，才回到票种检测列表重新选择可购买票种。
- 代理风控错误仍保持优先切换代理节点，切换成功后重新进入订单准备流程。
- `docs/api.md` 补充任务下发阶段的订单重试策略说明。

验收情况：

- 已新增/调整运行器测试，覆盖抢票阶段 1 次 prepare 搭配 4 次 createV2，以及回流创建失败吃满 4 次 createV2 后回到检测列表。
